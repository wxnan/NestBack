### 一、 架构哲学与技术栈选型

#### 1. 架构分层设计
*   **基石层 (必须离线可用)：** 本地 SQLite 数据库，承载所有物品、空间、统计逻辑。未配置任何第三方服务时，App 依然是一个完美的单机版收纳工具。
*   **插件层 (用户自备服务)：** AI 识图/聊天 (LLM API)、商品数据 (扫码 API)、灾难恢复、实时协同。插件层通过“我的-数据管理”动态接入，拔插不影响基石。

#### 2. 技术栈矩阵 (跨平台首选 Flutter)

| 模块 | 技术选型 | 理由 |
| :--- | :--- | :--- |
| **前端框架** | **Flutter** | 跨平台一致性，极客社区认可度高，丰富的本地处理与手势交互插件生态 |
| **本地数据库** | **SQLite + Drift (ORM)** | Drift 提供类型安全的 SQL 响应式查询，完美支持本地优先架构与复杂树形结构 |
| **实时同步** | **Supabase_flutter** | 动态初始化，支持 Realtime 订阅和 Postgres 行级安全 (RLS) 隔离多家庭数据 |
| **冷备灾备** | **Webdav_client + Encrypt** | 坚果云等兼容性好；配合 Encrypt 包做 AES 加密，保证网盘隐私 |
| **AI 接入** | **Dio (HTTP Client)** | 纯 REST API 调用，支持自定义 Base URL，兼容 OpenAI/国产大模型/本地 Ollama |
| **扫码识别** | **Mobile Scanner (ML Kit)** | 纯端侧识别条码，零延迟零成本；获取条码数字后调用用户配置的扫码 API |
| **语音输入** | **Speech_to_text** | 调用系统原生语音识别，免费，离线可用，用于 AI 聊天录入 |
| **手势交互** | **Flutter Slidable** | 完美实现左滑减一、右滑加一的高频快捷操作 |
| **图表统计** | **FL Chart** | 轻量级、高定制化的 Flutter 原生图表库，绘制统计饼图 |

### 二、 数据库核心设计 (Drift - 本地优先)

为了支撑多家庭、层级穿透、软删除（垃圾桶）和等待估值等复杂逻辑，数据库设计如下：

```dart
// 1. 房屋表 (多家庭支持)
class Houses extends Table {
  TextColumn get id => text()(); 
  TextColumn get name => text()();
  BoolColumn get isDefault => boolean().withDefault(const Constant(false)); // 标识是否有"待整理/垃圾桶"
}

// 2. 空间表 (无限层级穿透 + 特殊空间)
class Spaces extends Table {
  TextColumn get id => text()();
  TextColumn get houseId => text()(); // 所属家庭
  TextColumn get name => text()();
  TextColumn get parentId => text().nullable()(); // 父空间ID，null表示顶级(房间)
  TextColumn get type => text()(); // 枚举: 'room', 'container', 'sub_container', 'pending'(待整理), 'trash'(垃圾桶)
}

// 3. 物品表 (核心数据)
class Items extends Table {
  TextColumn get id => text()();
  TextColumn get houseId => text()(); // 所属家庭
  TextColumn get spaceId => text()(); // 所在空间 (可为pending或trash)
  TextColumn get name => text()();
  IntColumn get quantity => integer().withDefault(const Constant(1))();
  RealColumn get price => real().nullable()(); // 价格，null则为"等待估值"
  DateTimeColumn get expireDate => dateTime().nullable()(); 
  TextColumn get tags => text()(); // JSON数组存储标签
  TextColumn get creatorId => text()(); // 创建者 (用于按成员统计)
  TextColumn get modifierId => text()(); // 最后修改者
  DateTimeColumn get createdAt => dateTime()();
  DateTimeColumn get updatedAt => dateTime()(); // 同步冲突解决关键字段
}
```

**逻辑约束：**
*   每个家庭 (`houseId`) 创建时，系统自动生成两个不可删除的顶级空间：`type='pending'` (待整理) 和 `type='trash'` (垃圾桶)。
*   当物品 `quantity` 被操作减至 0 时，触发器或业务逻辑自动将其 `spaceId` 移至该家庭的 `trash` 空间，并将 `quantity` 重置为 1（恢复时可用）。

### 三、 页面布局与交互设计深度剖析

#### Tab 1：首页 (状态感知与极速寻物)

*   **家庭切换器：** 顶部左侧下拉菜单 `[🏠 我的家 ▾]`。切换后，首页、空间、统计所有数据查询瞬间切换为该家庭 ID。
*   **关键统计横幅：** 紧贴搜索栏下方，两张醒目卡片：
    *   🔴 **已经过期：** 点击直接跳转过期物品列表。
    *   🟡 **即将过期：** 点击跳转即将过期物品列表。
*   **搜索栏：** 支持拼音首字母、模糊匹配。本地 SQLite 查询，0延迟。
*   **分类筛选：** 横向滚动的 Chip 组（全部 | 食品 | 日用品 | 药品...）。选中状态高亮，列表无缝过渡。
*   **标签筛选：** 横向滚动的 Chip 组（全部 | 易碎 | 常用 | 出差...）。选中状态高亮，列表无缝过渡。如果没有标签，默认不显示标签筛选 Chip 组。
*   **物品列表：** 
    *   **排序切换：** 右上角图标切换 `最近添加` / `最近过期` / `最近修改`。
    *   **列表项 Card：** 左侧缩略图，中间名称+位置路径+库存数量，右侧过期倒计时天数。
    *   **滑动手势（快捷修改物品数量）：**
        *   **点击：** 点击物品卡片，进入物品详情页。
        *   **向右滑：** 显露 `[-1]` (消耗) 按钮和 `[+1]` (补货) 按钮。点击对应按钮，数量减1或加1并刷新；若减至0，移至回收站。已过期物品左滑直接显露 `[扔掉]` (移至垃圾桶)。移至回收站或垃圾桶操作需要确认弹窗。移至回收站或垃圾桶后，物品属性增加原始位置字段，用于记录移至回收站或垃圾桶前的位置。
        *   **向左滑：** 显露 `复制` (复制) 按钮和 `拆分` (拆分) 按钮。数量小于等于1时，不显示拆分按钮。
        *   **长按：** 支持多选，支持 `移动` / `删除` 操作。
*   **物品拆分：** 
    *   **核心思想：** 物理世界里的不同状态/不同位置的物品，本质上已经是“两个不同的物品”了。在数据库里，它们就是两条独立的 Item 记录。
    *   点击 `拆分` 按钮，进入拆分物品详情页。
    *   修改新物品名称、数量、位置、状态、标签等字段。数量需要小于物品当前数量（默认1），当填写的数量大于等于物品当前数量时，给出提示弹窗。
    *   确认后，新物品自动添加到数据库中，旧物品数量自动减少相应数量。
*   **物品复制：** 
    *   点击 `复制` 挌钮，进入复制物品详情页。
    *   修改新物品名称、数量、位置、状态、标签等字段。
    *   确认后，新物品自动添加到数据库中，旧物品数据保持不变。



#### Tab 2：空间 (收纳骨架树与高频操作)

*   **层级穿透：** 顶部面包屑导航 (`我的家 > 厨房 > 冰箱`)。列表展示当前层级下的【子空间卡片】+【物品卡片】。
*   **默认空间：** 每个家庭最顶层始终显示三个特殊卡片：`待整理` (放不知位置的物品) 、 `回收站`（放已删除的物品） `回收站-垃圾桶` （放已过期扔掉的物品）。
*   **添加空间：** 每个层级页面右下角常驻 `+ 添加空间`，弹出极简表单（名称、图标选择）。
*   **长按交互 (多选与移动)：**
    *   长按任意空间或物品进入多选模式。（不适用待整理、回收站和垃圾桶三个特殊空间）
    *   底部弹出操作栏：`移动` (弹出位置选择器，点选即移) / `删除`。
*   **回收站或垃圾桶特殊逻辑：** 进入回收站或垃圾桶空间后，物品左滑显露 `[删除]` (从DB抹去) 和 `[恢复]` (编辑后移回正常空间)。

#### Tab 3：录入 (魔法召唤阵)

中间突出的 `+` 号，点击后**从底部弹出半屏模态框**，展示四大入口。所有方式最终汇入**【统一确认表单】**，杜绝 AI 幻觉直接入库。

*   📷 **AI 识图：** 拍照 -> 压缩转 Base64 -> 发送至用户配置的 Vision API -> 返回 JSON 预填表单。
*   🎙 **AI 聊天：** 按住说话 -> 语音转文本 -> 发送至 Chat LLM (Function Calling) -> 返回结构化参数预填表单。
*   🔲 **扫码录入：** ML Kit 扫出条码 -> 发送至用户配置的扫码 API -> 返回商品信息预填表单。若 API 未配置/查无此码，退回手动/识图。
*   ✍ **手动录入：** 直接跳转物品编辑表单，预填表单。

*   **【物品编辑表单】：** 基本信息、扩展信息。
    *   **【基本信息】：** 封面（图片）、名称（必填）、分类（必填，从预设分类选择器中选择，默认"其他"）、位置（必填，默认当前所在空间）、数量（必填，步进器，默认1）、单位（如件、个、瓶等，默认"件"）、单价(可选，不填则归入"等待估值")、总价（可选，计算字段，数量*单价，单位：元）、标签（可选，多个标签用逗号分隔，如#易碎,#易冻）。
    *   **【扩展信息】：** 品牌、颜色、最低库存、储存方式、生产日期、保质期、过期日期、过期提醒、开封日期、购买日期、保修期、过保日期、条形码、创建日期、创建者、修改日期、修改者、备注等。不同的分类，不同的扩展信息，根据分类动态展示，均为可选字段。创建日期、创建者、修改日期、修改者是所有物品的通用字段，由系统自动维护。使用扫码录入方式时，条形码会自动填充到条形码字段中。
        *   **【食品类扩展信息】：** 品牌、颜色、储存方式、生产日期、保质期、过期日期、过期提醒、开封日期、条形码、备注等。
        *   **【药品类扩展信息】：** 品牌、颜色、储存方式、生产日期、保质期、过期日期、过期提醒、开封日期、条形码、备注等。
        *   **【美妆类扩展信息】：** 品牌、颜色、储存方式、生产日期、保质期、过期日期、过期提醒、开封日期、条形码、备注等。
        *   **【日用品类扩展信息】：** 品牌、颜色、最低库存、条形码、备注等。
        *   **【数码类扩展信息】：** 品牌、颜色、购买日期、保修期、过保日期、条形码、备注等。
        *   **【其他类扩展信息】：** 品牌、颜色、最低库存、条形码、备注等。

#### Tab 4：统计 (数据透视与断舍离驱动)

*   **顶部核心指标：** 6 个大数字卡片：物品总数、物品总价、已经过期、即将过期、库存不足 (数量<=阈值的消耗品)、等待估值 (price 为 null 的物品数)。
*   **物品列表：** 点击数字卡片，进入对应物品列表页面。
*   **统计饼图：** 使用 `fl_chart` 绘制交互式环形图。
    *   **维度切换：** 底部 Tab 切换 `按分类` / `按房间` / `按标签` / `按成员(创建者)`。
    *   **下钻交互：** 点击饼图扇区，在该分类下继续按某维度（分类、房间、标签、成员）筛选统计。
    *   **筛选结果：** 点击第1个饼图扇区或下钻交互的饼图扇区，筛选结果直接在饼图下方展示物品卡片列表。

#### Tab 5：我的 (极客控制台)

分为四大区块，严谨的逻辑分组：

- **区块一：账户与家庭 (协同中枢)**
    - 用户资料卡片：头像、昵称。
    - 家庭管理：切换/创建/删除（退出）家庭。
    - 成员管理：查看当前家庭成员及角色。
    - 邀请成员：生成邀请码 (包含 Supabase 连接信息与 HouseID)。
    - 加入家庭：输入邀请码，自动配置 Supabase 并加入。
- **区块二：物品管理 (业务规则)**

    - 属性管理：管理动态扩展属性 (如数码类增加"保修期")。
    - 标签管理：管理扁平标签 (如#易碎)。

### 四、 录入方式与照片同步实现逻辑

#### 1. 照片三层架构 (极客额度保护)
**绝对不要把高清原图直接存 Supabase！**
*   **本地层：** 拍照后压缩至 1MB 左右存本地沙盒，实现 0延迟展示。
*   **Supabase 层 (缩略图同步)：** 上传前二次压缩至 **30KB (200x200)** 缩略图存入 Supabase Storage。1GB 免费额度可存 3万张，足够全家使用。其他成员同步时先拉缩略图，保证列表不空白。
*   **WebDAV 层 (原图冷备)：** 每天凌晨将本地 1MB 原图打包加密上传至用户自备的坚果云。需看高清图时从 WebDAV 按需拉取。

#### 2. AI/扫码降级策略
*   **AI 识图/聊天：** 若用户未配置 AI Key，点击入口时弹窗提示："需在[我的-AI设置]配置 LLM API 后解锁魔法"。不阻塞手动录入。
*   **扫码：** ML Kit 本地极速解出条码数字。若未配置扫码 API，无法获取商品名，自动跳转至 AI 识图或手动录入，条码数字自动填入条形码字段。

### 五、 后端配置与接入设计 (BYOB 策略)

核心原则：**“基础同步服务由管理员统一承担，消耗性服务与个人备份由成员自行负责。”**

#### 1. Supabase (同步 - 管理员统一配置)
*   **配置归属：** 房屋管理员。
*   **加入机制：** 管理员配置完 Supabase 并运行一键 SQL 后，生成邀请码。成员输入邀请码，App **自动解析并写入 Supabase 配置**，成员无感知直连云端。
*   **RLS 隔离：** 提供的一键 SQL 脚本必须包含严格的行级安全策略，确保不同家庭的数据绝对隔离，成员只能看自己家庭的数据。

#### 2. AI 与扫码 API (消耗性服务 - 分层配置)
*   **配置归属：** 支持双重配置。
    *   **家庭级 (管理员)：** 管理员可配置"家庭公用 AI Key"，所有成员默认使用，管理员承担费用。
    *   **个人级 (成员覆盖)：** 成员可在 App 内配置"个人 AI Key"，一旦配置，优先使用个人的，实现费用自担与隐私保护。

#### 3. WebDAV (冷备 - 成员单独配置)
*   **配置归属：** 成员单独配置。
*   **逻辑：** 爸爸备份到坚果云，妈妈备份到阿里云盘。每个成员掌握自己数据的冷备副本，避免多人共用一个 WebDAV 导致文件覆盖冲突，也保证即使管理员跑路，成员仍有数据底座。

### 六、 开发路线图 (渐进式极客打法)

1.  **Phase 1：单机基石**
    *   搭建 Drift 数据库，完成多家庭切换、层级穿透、垃圾桶/待整理逻辑。
    *   完成手动录入与首页滑动手势。
2.  **Phase 2：数据透视**
    *   完成分类/标签/属性管理。
    *   完成统计页饼图与等待估值逻辑。
3.  **Phase 3：极客底座**
    *   完成“我的”页面四大配置项 UI 与连通性测试逻辑。
    *   实现 WebDAV 加密备份与恢复。
    *   编写保姆级 WebDAV 配置帮助文档。
4.  **Phase 4：AI 魔法解锁**
    *   接入 LLM API 配置与分层逻辑 (家庭公用+个人私有)。
    *   实现 AI 识图与聊天录入，汇聚至统一确认表单。
    *   接入扫码 API 配置与 ML Kit 本地扫码降级逻辑。
5.  **Phase 5：云端协同**
    *   接入 Supabase 动态初始化与 Realtime。
    *   生成一键建表 SQL 脚本与邀请码自动配置逻辑。
    *   实现照片缩略图同步策略。

**发布策略：** Phase 1+2 完成后即在 V2EX/GitHub 发布 v0.1 内测版，标题：*《一个纯本地、零服务器、极客专属的家庭收纳 App，自带 Key 解锁 AI 与云同步》*，吸引硬核用户打磨基础体验，再逐步迭代高级功能。
